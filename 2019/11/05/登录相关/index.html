登录相关

HTTP是无状态的协议，两次请求之间服务器并不会保存任何的数据
登录即用某种方法让服务器在多次请求之间能够识别出你
从登入到登出这个过程，服务器一直维持一个可识别出用户信息的数据结构，广义来说，该过程就叫session，即保持了一个会话

常见的两种登录

  1. 服务器session + 客户端sessionId
    client ———— /login POST username&password ————> server
    client <——— response 200 set-Cookie sessionId: xxx ——— server
    client ———— _api_userinfo GET cookie: sessionId=xxx ———> server

  2. token
    把用户的状态信息加密成一串token传给前端，请求时带上，服务器收到请求后，解析token并验证相关信息

业内通用加密方式 jwt (json web token)

放在HTTP header中
Authorization: Bearer

存在的问题

sessin：
1. 在服务端维护session信息，单机还好，多机还要在服务器间同步session信息，服务横向扩展不便
2. session 数量随登录用户增多而增多
3. session + cookie 里存sessionId的方式可能会有CSRF攻击，常见用csrf-token 解决

jwt：
1. 过期时间需结合业务作设置，且jwt一旦派发出去，后端无法强行使其作废

CSRF攻击

跨站域请求伪造

当前防御的几种策略

  1. 验证HTTP Referer字段
    http头部有个字段Referer记录了该HTTP请求的来源地址
    只需在最后给所有安全敏感的请求统一增加一个拦截器来检查Referer的值

缺点： Referer的值由浏览器提供，各浏览器实现不一
侵犯隐私，用户可以自行设置不发Referer
